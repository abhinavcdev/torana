Honest by design

What it does — and what it doesn't, yet.

Every config field is either enforced or warns loudly at startup. No silent ignoring, no feature claims the code can't back.

Shipping today

  • ✓HTTP/1.1 proxying with fully streamed bodies — no buffering
  • ✓Host- and path-based routing across multiple routes, first match wins
  • ✓TLS termination with rustls; pure Rust, no OpenSSL
  • ✓Upstream keep-alive pooling, TCP_NODELAY on both legs
  • ✓Weighted round-robin load balancing
  • ✓Active health checks — unhealthy upstreams are skipped, not blackholed
  • ✓Retries on connect failure for GET/HEAD/OPTIONS with no body
  • ✓Sandboxed WASM plugins, opt-in — fuel-limited, no host access at all
  • ✓Zero-downtime reload on SIGHUP, validated before swap
  • ✓Graceful shutdown — in-flight requests finish before exit
  • ✓Per-route timeouts — hanging upstreams return 504, not a stall
  • ✓Prometheus /metrics, /healthz, structured JSON logs
  • ✓Correct proxy headers — hop-by-hop stripped, X-Forwarded-* set
  • ✓Embeddable — the engine is a separate crate usable inside your own server

On the roadmap

  • —HTTP/2 and HTTP/3
  • —TLS to upstreams (https:// upstreams are rejected, not faked)
  • —Circuit breaking — retries and health checks exist; a faster-ejecting breaker doesn't yet
  • —Header rewriting, traffic mirroring, mTLS
  • —Automatic HTTPS — use Caddy if you want managed certs

Quick start

A minimal config: one HTTP listener, one route, one upstream.

torana.toml
[global]
log_level = "info"
metrics_addr = "127.0.0.1:9090"

[[listener]]
addr = "127.0.0.1:8080"
protocol = "http"

[[route]]
name = "default"
upstream = [{ addr = "http://127.0.0.1:3000" }]

Run it, send a request, check the metrics:

shell
./torana --config torana.toml &
curl -i http://127.0.0.1:8080/       # proxied to :3000
curl http://127.0.0.1:9090/metrics   # prometheus text

HTTPS termination

add a tls listener
./scripts/gen-certs.sh   # self-signed localhost pair

[[listener]]
addr = "0.0.0.0:8443"
protocol = "https"
tls_cert = "./certs/tls.crt"
tls_key = "./certs/tls.key"

Certificates are PEM; any CA-issued pair works the same way. There is no ACME — bring your own certs (or put Caddy in front if you want them managed).

Configuration

TOML, one file, validated at startup and on every reload. Invalid configs exit non-zero at startup and are rejected (keeping the running config) on reload. Fields marked planned parse without error so configs stay forward-compatible, but torana logs a warning rather than silently ignoring them.

Durations

Duration strings accept ms, s, and m: "500ms", "30s", "5m". A bare number is rejected — units are mandatory.

Weights

Weights are relative, reduced by their greatest common divisor: 90 / 10 produces the same schedule as 9 / 1. An upstream with no weight gets 100. The scheduler interleaves upstreams rather than sending long runs to one host.

Routing

Routes are matched in config order — first match wins. Put specific host/path combinations first and a constraint-free route last as a catch-all. A request that matches no route gets 404.

host + path routing
[[route]]
name = "api"
host = "api.example.com"
path = "/v1"
upstream = [{ addr = "http://127.0.0.1:3000" }]
retries = 2

[route.health_check]
path = "/healthz"
interval = "10s"
timeout = "2s"

[[route]]              # catch-all, no host/path — put it last
name = "default"
upstream = [{ addr = "http://127.0.0.1:4000" }]

Retries

Only GET, HEAD, and OPTIONS requests with an empty body are retried, and only on a connect-level failure — a request that reached the upstream is never retried, so a non-idempotent request is never double-executed. Retried attempts skip upstreams already tried this request and prefer upstreams the health check considers up.

Sandboxed WASM plugins

Build with cargo build --release --features plugins (off by default — wasmtime's Cranelift JIT roughly doubles the binary, from ~4.6 MB to ~11 MB). A plugin is a WASM module exporting memory, alloc(len) -> ptr, and on_request(method_ptr, method_len, path_ptr, path_len) -> i32: return 0 to allow, a code in 400..600 to deny with that status, anything else denies with 403. No host functions are exposed — a plugin cannot reach the filesystem, network, or process; wasmtime's sandbox is the entire boundary, and every call runs under a fuel limit so a runaway loop is denied with 500 instead of hanging the request.

attach a plugin to a route
[[route]]
name = "api"
plugin = "./plugins/auth-filter.wasm"
upstream = [{ addr = "http://127.0.0.1:3000" }]

Using it as a library

The proxy engine is a separate crate, torana-core, with none of the standalone binary's listener/signal machinery. Embed it inside a hyper server you already run instead of running torana as a separate process:

embed ProxyEngine
use torana_core::{Config, ProxyEngine, Metrics};
use std::sync::Arc;

let engine = ProxyEngine::new(config, Arc::new(metrics));
engine.spawn_health_probers().await;

// inside your own service_fn:
let response = engine.handle(req, client_addr, "http").await?;

Server (what the torana binary uses) is a thin wrapper around ProxyEngine that adds listener binding, SIGHUP reload, and graceful shutdown on top — read torana-core/src/server.rs if you want that full behavior instead of wiring it up yourself.

Signals & reloads

SignalBehavior
SIGHUPRe-read and validate the config file. On success, listeners keep running and new load-balancer schedules apply immediately. On failure, the running config is kept and the error is logged.
SIGTERM / SIGINTGraceful shutdown: stop accepting, let in-flight requests finish (15 s cap), then exit 0.
zero-downtime reload
# edit torana.toml, then:
kill -HUP $(pgrep torana)
Listeners themselves are bound once at startup — a reload updates routes, upstreams, and weights, but adding or removing a [[listener]] requires a restart. With graceful draining, a rolling restart behind an orchestrator is zero-dropped-requests anyway.

The metrics listener (global.metrics_addr) serves two paths: /metrics (Prometheus text format) and /healthz (returns 200 ok — wire it into liveness and readiness probes).

Logs

Structured JSON on stdout by default (log_format = "json"); set any other value for human-readable text. Levels follow log_level. Startup logs every bound listener; warnings fire for accepted-but-unimplemented config fields.

Deployment

docker-compose front door

compose.yaml
services:
  torana:
    build: https://github.com/abhinavcdev/torana.git
    ports: ["8080:8080"]
    volumes: ["./torana.toml:/etc/torana/torana.toml:ro"]
    depends_on: [app]
  app:
    image: your-app

Kubernetes sidecar

Build the image from the repo Dockerfile, push it to your registry, and mount the config from a ConfigMap. Probes hit /healthz; graceful draining makes rolling restarts drop zero requests.

pod spec (excerpt)
containers:
  - name: app
    image: your-app
  - name: torana
    image: your-registry/torana:0.4.0
    args: ["--config", "/etc/torana/torana.toml"]
    ports:
      - containerPort: 8080
    volumeMounts:
      - name: torana-config
        mountPath: /etc/torana
    livenessProbe:
      httpGet: { path: /healthz, port: 9090 }
    readinessProbe:
      httpGet: { path: /healthz, port: 9090 }
volumes:
  - name: torana-config
    configMap:
      name: torana-config
Expose metrics_addr as 0.0.0.0:9090 inside containers so probes and scrapers can reach it; keep it on 127.0.0.1 when the proxy shares a network namespace with its scraper.

It reports throughput, p50/p99 latency, idle memory after load, and image sizes. Docker on macOS runs in a VM — treat results as a relative comparison; absolute numbers are better on Linux bare metal.

Troubleshooting

torana exits immediately at startup

Startup is fail-fast by design. The error on stderr says which check failed: an unparseable config, a listener address already in use, missing TLS material, or a rejected field (for example an https:// upstream). Fix the config; torana never runs half-configured.

Clients get 502

502 Bad Gateway means the upstream connection failed — the upstream is down, refused the connection, or reset mid-request. Check upstream_connection_errors in metrics and the error log line naming the upstream.

Clients get 504

504 Gateway Timeout means the upstream accepted the connection but didn't return response headers within route.timeout.total (default 30s). Raise the timeout for slow endpoints or fix the upstream.

My reload didn't apply

If the new config fails validation, torana keeps the old one running and logs the reason at error level. Check the logs; fix; SIGHUP again. Also note listener changes need a restart (see Signals).

A config field seems ignored

It probably is — deliberately. Fields for unimplemented features log a startup warning like 'when' expressions are not implemented yet. The configuration table is the source of truth for what's enforced.

Roadmap

Route matching, active health checks, connect retries, and a sandboxed WASM plugin hook shipped in 0.4. In rough priority order from here: HTTP/2 (client-facing and upstream), TLS to upstreams, a circuit breaker that ejects a flapping upstream faster than health checks alone, header rewriting, traffic mirroring, and mTLS. The config schema already reserves fields for most of these — configs written today keep working.

If one of these matters to you, open an issue — early-stage projects reorder roadmaps for real users.
